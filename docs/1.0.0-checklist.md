# 1.0.0 Stabilization Checklist

This checklist defines what should be complete before promoting the project from the current pre-`1.0.0` state to a stable `1.0.0` release.

## Release Goal

`1.0.0` should mean the project is intentionally stable for public use, with well-defined CLI and API contracts, repeatable build steps, and documented operational expectations.

## 1. Versioning and Contract Freeze

- [ ] Confirm the intended `1.0.0` scope is complete
- [ ] Freeze the CLI argument structure
- [ ] Freeze the JSON output schema returned by `validate_semver()`
- [ ] Freeze the `/validate` response contract
- [ ] Freeze the `/analyze` request and response contract
- [ ] Decide which future changes would count as breaking after `1.0.0`
- [ ] Document those public contracts clearly in `README.md` and release notes

## 2. Core Validator Readiness

- [ ] Confirm SemVer 2.0.0 validation remains strict
- [ ] Re-check leading-zero rejection rules
- [ ] Re-check prerelease parsing behavior
- [ ] Re-check build metadata parsing behavior
- [ ] Re-check malformed/incomplete version rejection behavior
- [ ] Ensure `validate_semver()` remains the single source of truth for validation
- [ ] Ensure the function always returns JSON-serializable dictionaries

### Suggested validation spot checks

Should pass:
- [ ] `1.0.0`
- [ ] `0.1.0`
- [ ] `2.1.3-alpha`
- [ ] `1.0.0-beta.1`
- [ ] `1.0.0+build.1`
- [ ] `1.0.0-beta.1+exp.sha.5114f85`

Should fail:
- [ ] `01.0.0`
- [ ] `1.02.3`
- [ ] `1.0`
- [ ] `1.0.0-`
- [ ] `1.0.0+`
- [ ] `1.0.0-alpha..1`

## 3. CLI Stability

- [ ] Verify `semver-cli` installs correctly from the built package
- [ ] Verify valid versions produce the expected JSON output
- [ ] Verify invalid versions produce failure JSON
- [ ] Verify invalid CLI validation exits with status code `1`
- [ ] Verify help text is correct and stable
- [ ] Verify interactive mode still works as intended
- [ ] Document the supported CLI usage patterns in `README.md`

## 4. API Stability

- [ ] Verify `semver_validator.api` imports cleanly
- [ ] Verify `/validate` handles valid input correctly
- [ ] Verify `/validate` handles invalid input correctly
- [ ] Verify `/analyze` validates `current_version` before calling Gemini
- [ ] Verify `/analyze` returns a stable JSON structure on success
- [ ] Verify `/analyze` returns clear error responses on backend/upstream failures
- [ ] Verify no secret values are exposed in API responses
- [ ] Document the final request/response format for both endpoints

## 5. Gemini Analyzer Readiness

- [ ] Test `/analyze` locally with a real `GEMINI_API_KEY`
- [ ] Verify `.env` is loaded from the project root
- [ ] Verify `.env_example` matches the expected environment variables
- [ ] Confirm the frontend never needs direct access to the Gemini API key
- [ ] Confirm Gemini failures return safe, explicit errors
- [ ] Confirm analyzer output is structured enough for frontend use
- [ ] Decide whether retry logic or stricter response validation is needed before `1.0.0`

## 6. Automated Test Coverage

- [ ] Keep validator tests passing
- [ ] Keep CLI tests passing
- [ ] Keep API tests passing
- [ ] Keep analyzer endpoint tests passing
- [ ] Add any missing edge-case tests discovered during manual verification
- [ ] Run the full test suite from a clean environment
- [ ] Decide whether CI should be added before the `1.0.0` release

## 7. Packaging and Build Readiness

- [ ] Verify `pyproject.toml` version is updated to `1.0.0` only when ready
- [ ] Verify runtime dependencies match actual imports
- [ ] Verify test dependencies are kept separate from runtime dependencies
- [ ] Build both sdist and wheel successfully
- [ ] Install the built wheel in a clean virtual environment
- [ ] Run CLI smoke tests from the installed wheel
- [ ] Confirm package metadata and entrypoints are correct

## 8. Documentation Readiness

- [ ] Review `README.md` for current and accurate setup instructions
- [ ] Review API examples for correctness
- [ ] Review analyzer setup instructions for correctness
- [ ] Review `AGENTS.md`, `CODEX.md`, and `GEMINI.md` for current guidance
- [ ] Review `INSTRUCTIONS.md` for current release guidance
- [ ] Ensure all filename references are current and conventional
- [ ] Add release notes for `1.0.0`

## 9. Branching and Release Process

- [ ] Create `release/1.0.0` from `master`
- [ ] Merge all required `feat/...`, `fix/...`, `docs/...`, and `chore/...` branches
- [ ] Re-run tests and build verification on the release branch
- [ ] Finalize version number and docs on the release branch
- [ ] Merge `release/1.0.0` back into `master`
- [ ] Tag the release on `master` as `v1.0.0`

## 10. Public Release Readiness

- [ ] Decide where the API will be hosted publicly
- [ ] Confirm environment variable management for deployment
- [ ] Confirm production startup command and runtime expectations
- [ ] Confirm any frontend integration points are documented
- [ ] Confirm the public API is stable enough to support external users
- [ ] Confirm error messages are helpful without leaking implementation details

## Final Go/No-Go Check

Before cutting `1.0.0`, confirm:

- [ ] the CLI is stable
- [ ] the API is stable
- [ ] the analyzer is safe and usable
- [ ] the build is repeatable
- [ ] the docs match the code
- [ ] the public contracts are intentional
- [ ] the team is comfortable treating future breaking changes as `2.0.0` work
